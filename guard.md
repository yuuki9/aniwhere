# Guard

이 프로젝트의 프론트엔드는 Apps in Toss WebView 미니앱을 기준으로 작업합니다.

## 기본 원칙

- 공식 기준 문서는 항상 Apps in Toss 문서를 우선 참고합니다.
- 화면 구조는 Toss 스타일의 정보 탐색형 UX를 기본으로 합니다.
- API 구조를 따르되, UI는 단순 관리 화면처럼 보이지 않게 다듬습니다.
- 지도는 메인보다 탐색 보조 수단으로 사용합니다.

## 현재 프로젝트 해석

- 프론트 위치: `client/`
- 프레임워크: `@apps-in-toss/web-framework`
- 설정 파일: `client/granite.config.ts`
- 게임형 앱이 아니라 서비스형 WebView 앱으로 간주합니다.

## UX 방향

- 첫 화면은 소개보다 실제 탐색 진입이 우선입니다.
- 기본 진입은 샵 탐색 화면과 커뮤니티 흐름으로 연결합니다.
- 카드, 상태값, 검색, 필터가 즉시 읽히는 구조를 우선합니다.
- 모바일 기준으로 넓은 터치 영역과 짧은 의사결정 흐름을 유지합니다.

## 구현 방향

- `shops`, `community(posts/comments)`를 중심으로 화면을 구성합니다.
- Swagger 스펙 변경 시 프론트 타입과 API 호출 함수를 우선 맞춥니다.
- `brand.displayName`, `appName` 등 앱 메타 정보는 실제 서비스 명칭과 일치시킵니다.
- 출시 전에는 Sandbox 기준 검증과 실제 배포 설정을 다시 확인합니다.
