# AgentMemory 운영 가이드

이 문서는 Cursor 에이전트의 `agentmemory`를 팀 공용으로 사용할 때의 기준을 정리합니다.

## 목적

- 세션 간 의사결정 맥락을 유지합니다.
- 반복 질문/반복 작업 비용을 줄입니다.
- 운영 시 중요한 합의사항을 재사용 가능한 지식으로 축적합니다.

## 언제 저장할까

- 재사용 가치가 높은 결정이 나왔을 때
- 특정 계정/리전/보안 정책 같은 운영 고정값이 확정됐을 때
- 장애 원인, 재현 조건, 우회 방법이 명확해졌을 때

## 무엇을 저장할까

- `architecture`: 시스템/운영 구조 관련 결정
- `workflow`: 팀 표준 절차, 승인 플로우
- `preference`: 팀 선호 설정 (예: 기본 리전)
- `bug`: 장애 원인과 해결 패턴
- `fact`: 변하지 않는 사실성 정보

## 저장 권장 예시

- 기본 AWS 리전은 `ap-northeast-2` 사용
- API MCP는 기본 `READ_OPERATIONS_ONLY=true` 유지
- 변경 작업 전 dry-run 스타일 계획 먼저 검토

## 조회/활용 원칙

- 작업 시작 시 관련 키워드로 과거 메모리를 먼저 조회합니다.
- 조회 결과는 최신 정책 문서(`.cursor/mcp/shared/docs/ops-policy.md`)와 충돌 여부를 확인합니다.
- 충돌 시 문서 기준을 우선하고, 메모리를 갱신합니다.

## 거버넌스

- 민감정보(Access Key, Secret, 토큰, 개인식별정보)는 저장 금지
- 오래된 정책/폐기된 정보는 삭제 또는 정정
- 중요 변경은 문서와 메모리를 함께 업데이트

## 태그 기반 세션 저장 규칙

- 자동 저장은 사용하지 않고, 완료 태그를 저장 트리거 신호로만 사용합니다.
- 완료 태그는 `#task-done` 하나만 사용합니다.
- 세션 전환 태그는 `#new-session` 하나만 사용합니다.
- 완료 태그가 감지되면 먼저 저장 범위를 확인한 뒤 `agentmemory-memory_save`를 실행합니다.
- `#task-done` 감지 시 요약 저장 후 `#new-session` 입력을 안내합니다.
- `#new-session` 감지 시 현재 범위를 요약 저장한 뒤 다음 작업/새 세션으로 이동합니다.
- 전환 태그에서는 agentmemory의 "논리 세션" 마커를 먼저 저장합니다.
  - 예: `session-YYYYMMDD-HHmmss`
  - 마커 저장 형식: `type=workflow`, `concepts=session,session-open,<session_id>`
- 태그는 채팅 UI의 물리 세션을 자동 생성하지 않으며, 사용자 또는 에이전트가 새 채팅을 시작해야 합니다.
- 저장 내용은 아래 최소 항목을 권장합니다.
  - Goal
  - Key changes
  - Decision rationale
  - Next step

### 왜 자동 저장을 막는가

- 대화 단락 기준 자동 저장은 노이즈 메모리를 빠르게 늘립니다.
- 미확정 가설, 임시 아이디어, 민감정보가 저장될 위험이 있습니다.
- 회수 품질을 위해 "완료 태그 + 확인 후 저장" 원칙을 유지합니다.
